Problem Statement
No integrity mechanism at the IoT edge

CCTV-based smart parking systems capture and transmit video without any cryptographic verification. An attacker with access to the local network or storage can silently inject fake frames, delete real frames, or alter metadata — and existing systems have no way to detect it. Raw video transmitted to the cloud further introduces MITM risk and violates user privacy.

Proposed Solution
Lightweight SHA-256 hash chain on Jetson Nano

Every captured frame is cryptographically hashed using SHA-256. Each hash is chained to the previous, forming a tamper-evident ledger that runs entirely on the Jetson Nano edge device. Only hash records — never raw video — are anchored to AWS DynamoDB for remote verification. Video stays on-device by design.

Research Objectives
Four objectives

RO1 — Design a 3-layer security framework (edge detection, hash chain, cloud anchor) for IoT parking.

RO2 — Implement a lightweight blockchain on resource-constrained hardware without consensus overhead.

RO3 — Validate against 10 attack scenarios with statistical rigour (n≥30, CLT, 95% CI).

RO4 — Quantify performance overhead and prove system sustainability over 24 hours.

Key Contribution
Edge-first integrity without consensus

Traditional blockchain requires network consensus (proof-of-work, proof-of-stake) making it infeasible on constrained hardware. This work contributes a hash chain variant — cryptographically equivalent integrity without consensus — validated specifically for parking video streams on ARM Cortex-A57 (Jetson Nano).

The cloud anchor (DynamoDB) acts as a trusted external witness, not a consensus node, enabling replay attack detection without full blockchain infrastructure.

Implementation

Block Structure & Chaining Formula

frame_hash = SHA-256( raw_frame_bytes )

chain_hash = SHA-256(
  str(index) + "|" +
  str(timestamp) + "|" +
  frame_hash + "|" +
  prev_hash
)

Genesis block: prev_hash = "0" × 64
import hashlib, time from dataclasses import dataclass @dataclass class Block: index: int timestamp: float detection_result: str # "Occupied" | "Empty" frame_hash: str # SHA-256 of raw frame prev_hash: str # previous block's chain_hash chain_hash: str # integrity of this block def hash_frame(frame_bytes: bytes) -> str: return hashlib.sha256(frame_bytes).hexdigest() def compute_chain_hash(b: Block) -> str: data = f"{b.index}|{b.timestamp}|{b.frame_hash}|{b.prev_hash}" return hashlib.sha256(data.encode()).hexdigest()

Chain Verification Logic

Tamper at block k means:
block[k].chain_hash ≠ SHA-256(block[k].data)
block[k+1].prev_hash ≠ block[k].chain_hash

Both conditions checked per block → O(n) total
Break position pinpointed exactly at k
def verify_chain(chain: list[Block]) -> dict: # Verify genesis block itself if compute_chain_hash(chain[0]) != chain[0].chain_hash: return {"valid": False, "break_at": 0, "type": "genesis_tampered"} for i in range(1, len(chain)): b, prev = chain[i], chain[i - 1] # 1. Linkage: does this block reference prev correctly? if b.prev_hash != prev.chain_hash: return {"valid": False, "break_at": i, "type": "chain_break"} # 2. Integrity: is this block's own hash correct? if b.chain_hash != compute_chain_hash(b): return {"valid": False, "break_at": i, "type": "hash_mismatch"} return {"valid": True, "blocks_verified": len(chain)}

Cloud Anchor Upload

import boto3 dynamodb = boto3.resource('dynamodb', region_name='us-east-1') table = dynamodb.Table('parking_blockchain') def upload_anchor(block: Block): # Hash only — raw frame never leaves edge table.put_item(Item={ 'anchor_id': f"anchor_{block.index}", 'block_index': block.index, 'chain_hash': block.chain_hash, 'timestamp': block.timestamp, 'device_id': 'jetson-nano-01' }) def verify_against_cloud(block: Block) -> bool: resp = table.get_item( Key={'anchor_id': f"anchor_{block.index}"}, ConsistentRead=True ) item = resp.get('Item') if not item: return False return item['chain_hash'] == block.chain_hash

INTACT CHAIN — each block correctly references previous chain_hash
TAMPERED CHAIN — Block #2 pixel data modified → chain_hash invalid → break detected immediately at verification
Key property: Even a 1-pixel change in frame data produces a completely different SHA-256 hash (avalanche effect). The attacker cannot predict or reproduce the original hash without the original data. Chain breaks are localised — only the tampered block and everything after it becomes invalid.
Threat Model & Attack Mitigations
Threat model scope: Attacks on stored frames and chain records on the Jetson Nano (post-capture). Network-level attacks (MITM on RTSP) and physical camera tampering are out of scope — the hash chain validates what was recorded, not the network path.
AV-01 Frame Modification MITIGATED
Attacker modifies pixel data in a stored JPEG frame (any number of pixels, including just 1). SHA-256 avalanche effect guarantees a completely different frame_hash. The block's chain_hash then fails recomputation during verify_chain(). Attack is detected in O(1) per block.
Experiment E1 — Expected: 100% detection rate, n=30, <0.35 ms latency
AV-02 Frame Deletion MITIGATED
Attacker deletes a block from the chain. The successor block's prev_hash no longer matches the preceding block's chain_hash, creating a detectable gap at the exact deletion position. Detection latency is proportional to position in chain (blocks near start: ~0.15 ms, near end: ~1.6 ms).
Experiment E2 — Expected: 100% detection, latency varies by position
AV-03 Frame Injection MITIGATED
Attacker inserts fake frames — from another camera, AI-generated, or duplicated with altered metadata. The injected block cannot produce a valid chain_hash that simultaneously satisfies both its own integrity check and the linkage to adjacent blocks. Detection is content-agnostic.
Experiment E3 — Expected: 100% detection regardless of fake frame source
AV-04 Metadata Tampering MITIGATED
Attacker alters non-pixel fields: timestamp (+/- seconds), detection_result (Occupied→Empty to hide vehicle), or device_id. All metadata is included in chain_hash = SHA-256(index|timestamp|frame_hash|prev_hash). Any metadata change invalidates chain_hash immediately.
Experiment E4 — Expected: 100% detection across timestamp/result/device changes
AV-05 Hash Record Alteration MITIGATED
Attacker directly modifies stored hash values (single character change, full random replacement, or block swap). The recomputed chain_hash will not match the stored value. Altering chain_hash without modifying the underlying data is the most direct attack, but trivially detected.
Experiment E5 — Expected: 100% detection, even 1-character hex changes
AV-06 Replay Attack MITIGATED
Attacker replays old chain segments (4 scenarios: original timestamps+hashes, updated timestamps, recomputed hashes, both). Scenarios A–C detected via hash/timestamp inconsistency. Scenario D (hardest — full recompute with updated timestamps) detected via cloud anchor checkpoint mismatch.
Experiment E6 — Expected: ≥95% across all 4 scenarios (n=30 trials)
AV-07 Chain Rewrite MITIGATED
Attacker rewrites K blocks, recomputing all chain_hashes. Rewrite cost = O(K × SHA-256). On Jetson Nano (ARM), 1,000 blocks ≈ 1.14 sec rewrite time vs 0.16 sec on desktop. The constrained hardware makes large rewrites expensive. Cloud anchor provides independent reference that cannot be rewritten by local attacker.
Experiment E7 — Expected: cloud detects within 1 sec; linear R² > 0.99
AV-08 Feed Substitution PARTIAL
Attacker replaces camera with loop/pre-recorded footage. Hash pattern analysis detects repeated frames (identical frame → identical frame_hash appears in chain). Lighting inconsistency provides secondary signal. Detection varies by time of day and substitution quality — night is easiest (90%), afternoon hardest (23% repeated hash rate).
Experiment E8 — Expected: ≥90% overall; partial gaps acknowledged as limitation
AV-09 Timestamp Manipulation MITIGATED
Attacker manipulates system clock (forward, backward, random date) to forge temporal context. The cloud anchor stores its own server-side timestamp at upload time. A local clock change creates a mismatch between the block's timestamp and the DynamoDB anchor record, detected in ≈915–930 ms (network roundtrip included).
Experiment E9 — Expected: ≥95% detection via cloud timestamp cross-check
AV-10 Physical Sensor Spoofing LAYER 1 SCOPE
Objects placed in spots to fool Canny detector: white poster, dark tape car outline, cardboard box, reflective surface, plastic bag, tree branch. This is a Layer 1 detection accuracy problem — not a security attack on the hash chain. The blockchain faithfully records whatever Canny reports, demonstrating that data integrity is independent of sensor accuracy. Edge count threshold analysis identifies spoofing objects (>800 edges tend to false-positive).
Experiment E10 — Characterises Layer 1 FP rate; not a blockchain failure

Statistical Methodology

CLT Compliance — Why n ≥ 30

The Central Limit Theorem states that sample means approach a normal distribution as n increases, regardless of population distribution. n ≥ 30 is the accepted minimum for parametric tests to be valid.

x̄  = Σxᵢ / n                  (sample mean)
s   = √[Σ(xᵢ − x̄)² / (n−1)]  (sample SD)
SE  = s / √n                  (standard error)
CI  = x̄ ± 1.96 × SE          (95% confidence interval)
